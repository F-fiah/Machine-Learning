# Debugging learning algorithm

When the algorithm doesn't work well, what to do next?
- get more data
- try a larger/smaller set of features
- try change the feature
- change learning rate
- ……
## Bias and Variance Diagnostic

Suppose you suspect the problem is either : 
- Overfitting (high variance)
- Too few features to classify spam (high bias)

 Diagnostic : 
 - Variance: Training error will be much lower than test error
 - Bias: Training error will also be high

![[variance.png]]
当 data set 增加时，training error 和 test error 的差距会缩小，但仍存在较大的差距
这中情况就表明模型存在方差问题

## Optimization Algorithm

Problem set :
- 算法是否合适（是否收敛了）
- 训练的指标和真正的业务指标之间存在区别
- 惩罚系数、正则超参等设置得是否合理

1. Case 1
    $a(\theta_{SVM}) > a(\theta_{BLR}),\quad J(\theta_{SVM}) > J(\theta_{BLR})$ 
	问题出在**优化算法/收敛**：
	BLR 的任务是最大化$J(\theta)$，但 SVM 的参数代入后$J$更高，说明梯度上升未收敛，没找到$J$全局最大值
	解决方案：
	- 调学习率
	- 增加迭代次数
	- 特征归一化
	- 监控$J$曲线至平缓
2. Case 2
    $a(\theta_{SVM}) > a(\theta_{BLR}),\quad J(\theta_{SVM}) \le J(\theta_{BLR})$ 
	问题出在**目标函数本身**：
	BLR 已成功最大化自身目标 $J(\theta)$ ，但最优 $J$ 对应的业务加权正确率依然不如 SVM ；代表$J(\theta)$作为代理损失，和业务指标$a(\theta)$优化方向脱节
	解决方案：
	- 使用加权对数似然，给高代价样本赋予权重
	- 训练后遍历阈值，选取最大化 $a(\theta)$ 的分界值
	- 交叉验证以验证集 $a(\theta)$ 筛选正则超参 $\lambda$
## Optimization Objective

Steps :
1. build a simulator
2. choose a cost function
3. use reinforcement learning algorithm to minimize the cost function

But the result is not good, we suppose to :
- improve simulator
- modify cost function
- modify the learning algorithm

Diagnostics: 
1. If $\theta_{\text{RL}}$ flies well in simulation, but not in real life, then the problem is in the simulator. 
   Otherwise:
2. Let $\theta_{\text{human}}$ be the human control policy. If $J(\theta_{\text{human}}) < J(\theta_{\text{RL}})$, then the problem is in the reinforcement learning algorithm. (Failing to minimize the cost function $J$)
3. If $J(\theta_{\text{human}}) \ge J(\theta_{\text{RL}})$, then the problem is in the cost function. (Maximizing it doesn’t correspond to good autonomous flight)
# Error Analysis

1. 把**某一个模块的输出强行替换成完全正确的真值（ground-truth）**
2. 固定其余所有模块不变，重新跑完整流程，记录系统整体准确率提升幅度
3. 提升越大，代表这个模块是当前最大误差来源，优先优化
# Ablative Analysis

1. 记录完整系统基线效果（所有组件/特征全部启用）
2. 遍历每一个独立组件/特征组： 
	- 仅删除当前这一组，其余全部保留不变
	- 运行模型，记录当前准确率/指标
3. 对比各组删除后的指标跌幅：跌幅越大，该组件贡献越高
4. 定位性能提升的核心模块，可删减低贡献组件
